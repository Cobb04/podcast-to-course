# Framework: Harness 三层架构

**Origin:** 新路（CRAI/1KB 创始人），十字路口播客 | 🟡 Medium（生产级 agent 开发经验）
**Related:** Learn Cloud Code 开源教程（GitHub 5万+ 星）

---

## When to Use

你在设计、评估或优化一个 agent 系统时，需要判断"哪里做得不够好"或"竞品卡在哪一层"。

---

## The Framework

```
┌──────────────────────────────────────────┐
│  第三层：编排治理层 (Orchestration)         │
│  · 多 Agent 协调与任务分配                  │
│  · 权限治理（谁能访问什么）                  │
│  · 上下文接力机制（上一个 agent → 下一个）    │
│  · 并行 vs 串行决策                        │
│  · Agent 的"组织架构"设计                   │
├──────────────────────────────────────────┤
│  第二层：上下文状态层 (Context)             │
│  · System prompt / Skills                │
│  · Memory（记忆的存储、更新、检索）          │
│  · 上下文卸载策略（窗口满了怎么办）           │
│  · 工作环境信息（路径、依赖、Git 状态）       │
│  · Agent 交接时的状态传递                   │
├──────────────────────────────────────────┤
│  第一层：执行能力层 (Action)               │
│  · CLI 工具（命令行接口）                   │
│  · 文件系统操作（增删读写搜索）              │
│  · 浏览器操作能力                          │
│  · 语言解释器（Python/Node）               │
│  · MCP 扩展工具                            │
└──────────────────────────────────────────┘
```

---

## Decision Rules

### 做产品时

| 你的情况 | 重点投入 |
|---|---|
| 单个 agent，简单任务 | **第一层** — 配好 3 类核心工具（文件系统、浏览器、解释器） |
| 单个 agent，长任务（超过上下文窗口） | **第一层 + 第二层** — 工具 + 上下文卸载/接力机制 |
| 单个 agent，多 session 用户 | **第二层为主** — Memory 和 Skills 的持续积累 |
| 多 agent 系统 | **三层都需要**，但第三层决定成败 |
| 开放平台（第三方 agent） | **第三层为主** — 编排、治理、沙箱隔离 |

### 评估竞品时

- **只做了第一层的 → Demo。** 能跑通一个任务，但不能持续工作、不能跨 session。
- **做到第二层的 → 产品。** 有记忆、有上下文管理，能处理长任务。
- **三层都做透的 → 平台。** 能支撑多 agent 协作、有权限体系、有治理能力。

---

## When NOT to Use

- **单次问答场景** — 不需要完整 harness。一个 API call 就够了。
- **模型能力不够时** — 如果基础模型任务成功率 < 60%，先换模型再优化 harness。
- **固定流程自动化** — 如果输入输出完全确定，传统工程方案比 agent 更可靠、更便宜。

---

## Boundary Conditions

新路指出这个框架有两个隐含前提：

1. **假设你在用 SOTA 或次 SOTA 的 agent 模型。** 如果是弱模型，第一层工具再多也跑不起来。
2. **假设任务复杂度需要三层。** 很多任务只需要第一层。不要过度设计 — 一个简单的 `grep` + `sed` 组合可能比多 agent 编排更有效。

---

## Credibility Calibration

- 🟡 这个框架来源于新路的工程实践和 Cloud Code 源码分析，**不是学术研究产出**。它更像一个实用的分类工具而非严格的理论模型。
- 🟡 三层之间的边界在实践中是模糊的。Memory 和 Skills 的边界尤其不清晰 — 新路自己也承认"中间的交集还蛮大的空间"。
- 🔴 "三层都做透的是平台"这个判断是新路的创业视角，可能与用户实际需求有偏差。很多成功的 agent 产品并没有做透三层。
