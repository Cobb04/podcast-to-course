两个可复用框架：三层架构 + 双自洽标准

这是整期播客最硬核的产出——两个你可以立刻拿去用的判断框架。

框架 1：Harness 三层架构

任何 agent 产品的技术架构，都可以拆成这三层来评估：

🔧 第一层：执行能力层

CLI工具 · 文件系统 · 浏览器 · 语言解释器

🧠 第二层：上下文状态层

Memory · Skills · 上下文卸载 · Agent接棒

🏛️ 第三层：编排治理层

多Agent协调 · 权限控制 · 沙箱隔离 · 接力机制

🎯 竞品评估只看做到第几层。第一层 = Demo，第二层 = 产品，第三层 = 平台。你现在的产品卡在哪一层？

三层架构的决策规则

你的产品阶段重点投入
单 Agent + 简单任务第一层 — 配好 3 类核心工具
单 Agent + 长任务（超上下文窗口）第一层 + 第二层 — 工具 + 上下文卸载/接力
单 Agent + 多 Session 用户第二层为主 — Memory 和 Skills 积累
多 Agent 系统三层都需要，但第三层决定成败
开放平台（第三方 Agent）第三层为主 — 编排、治理、沙箱
🎯
评估竞品时：只做到第一层的是 Demo。做到第二层的是产品。三层都做透的是平台。你现在的产品目标在哪个阶段？

框架 2：双自洽标准

这是新路判断任何 agent 技术方案会不会被淘汰的检验标准：

新路
"好的 harness 首先要符合模型本身在运行上的逻辑，其次要符合模型未来能力进步的逻辑。只要跟这两个点不 match 的，就是不好的 harness。"
🔑 双自洽检验法

自洽 1（原理层）：这个设计是否与模型运行原理自洽？（模型是自回归的 → 应自主决策下一步）

自洽 2（时间层）：如果模型变强 10 倍，这个设计会变得更有用还是变成瓶颈？

何时使用：每次技术选型时——尤其当工程师说"我们用 LangChain 的 prompt flow 来做"的时候。

用双自洽标准检验 LangChain Prompt Flow

工程师提出用 LangChain 的 prompt flow 模式——预设每个节点的输入输出，用状态图控制流转。用双自洽标准检验这个方案：它违反了哪条？

新路的判断是：Prompt flow 预设每一步的决策 → 违背自洽 1（模型应该自主决策下一步）。且模型越强，预设流程越成为瓶颈 → 违背自洽 2。这就是为什么他说这套方法论'会越来越不适用'。
