新路的思维模型：从"模型视角"理解 Harness

大多数开发者从"怎么让代码好维护"出发设计 agent 系统。新路从"模型怎么工作"出发。这是两种完全不同的思维方式。

模型的"母语"是什么？

新路
"如果你稍微思考一下 transformer 模型是怎么自回归的、推理是怎样运行的、agent 模型到底怎么工作的……你再结合这个来思考所有 agent 的最佳实践，你就觉得非常 make sense。"
🔑 可操作洞见

新路的所有判断都来自一个底层信念：工程应该适配模型，而不是让模型适配工程。

何时使用：每次技术评审时问——"这个设计是从模型视角出发的，还是从开发者便利出发的？"

两种派系的战争

当前 agent 开发存在两条路线。新路站哪边，一目了然：

Prompt Flow 派（LangChain / LangGraph）

我们预设好每个节点的输入输出，用状态图控制流转。这样可控、可调试、成本低。

Agent-Native 派（Cloud Code / 新路）

你预设的每一步都是在限制模型。模型是自回归的——它应该自己决定下一步做什么。你的"可控性"是模型能力的上限。

Prompt Flow 派

但完全放开的话，模型乱调用工具怎么办？成本和延迟怎么控制？

Agent-Native 派

More context, less control。给它充分的上下文和工具，限制只在于"某些工具不能调"，而不是"你必须按我的顺序调"。模型越强，你的预设流程反而越成为瓶颈。

⚠️
可信度校准：新路的立场非常鲜明——他正在做一个 agent-native 的工具链创业公司（1KB/CRAI）。他对 LangChain 的批判有工程实践支撑（🟡），但也带有创业者的竞争立场。他的框架很有用，但不要把它当作"已被验证的行业共识"。

但核心洞见是硬的

不管你是不是站新路这边，他的底层信念——"工程的目的是放大模型的能力，而不是限制它"——是对所有 agent 产品人都适用的检验标准。
