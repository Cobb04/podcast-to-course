Agent 的上限，到底由什么决定？

最近大家都在讨论 agent harness。但你有没有想过——当我们聊 harness 的时候，我们到底在聊什么？

💡
这期播客的核心问题：如果 agent 的上限是由 harness 决定的，那 harness 到底是什么？作为产品人，你不需要自己写 harness，但你必须能判断一个 harness 是好是坏。
🧠 思维模型新路的底层信念：工程应该适配模型，而不是让模型适配工程。这个比喻是整期播客的"根隐喻"——后面所有判断都从这里长出来。

一个比喻，打开整个话题

新路
"模型以外都是 harness。把模型比成人的话，我们智商在 120 到 170 之间——但我们可以通过健身、学舞蹈、修习武术、穿上机甲来让自己更加强大。机甲就是 harness。"
🔑 可操作洞见

模型决定智力天花板，harness 决定这个智力能发挥多少。

何时使用：每次评估 agent 产品时，先问"是模型不行还是 harness 不行？"——这是两个完全不同的优化方向。

为什么现在聊这个？

因为模型之间的差距正在缩小。当大家都用同一个 SOTA 模型时，模型不再是差异化的来源——harness 才是。

⚡
技术优先级 ≠ 商业优先级。技术上：模型 > 上下文 > 工具。商业上正好反过来——因为模型大家用的都一样，harness 才是护城河。

这期嘉宾 新路（CRAI/1KB 创始人）做了一件事：把 Cloud Code 泄露的源码拆开，分析了世界最顶尖的 agent harness 是怎么设计的。他的结论可能颠覆你对 agent 产品的理解。

自测：你的直觉对吗？

你是一个 AI PM。你的 agent 产品任务成功率只有 55%。工程师说"我们需要优化 harness——加更多工具、更好的上下文管理。"你的第一反应应该是什么？

模型才是 agent。如果基础成功率不到 60%，换更强的模型是性价比最高的第一步。harness 是放大器——信号太弱的话，放大也没用。
