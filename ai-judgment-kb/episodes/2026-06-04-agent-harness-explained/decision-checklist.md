# Decision Checklist: Agent 产品技术选型

**Derived from:** 十字路口 × 新路 — Agent Harness 深度拆解
**Use when:** 评估技术方案、选工具链、设计 agent 产品架构

---

## 选模型时的 3 个问题

- [ ] **当前 SOTA agent 模型是谁？** 新路的判断：SPIC（SOTA），Kimi/MiniMax/GLM 最新代（次 SOTA）。模型是第一决策，选错了后面都白做。
- [ ] **我的任务在这个模型上的裸成功率是多少？** 如果 < 60%，先别折腾 harness，换模型或等下一代。
- [ ] **模型是 agent-native 的吗？** （即专门为长程任务训练过）如果不是，它可能只能做单轮工具调用，做不了多步自主决策。

---

## 选工具链时的 4 个问题

- [ ] **用了 MCP 还是 CLI？** 优先 CLI。Unix 命令在预训练语料中充分得多，任务成功率更高。如果必须用 MCP，做 A/B 对比测试。
- [ ] **工具链设计是否与 agent 角色绑定？** 只读任务只配无副作用工具（禁止删改）。写代码的 agent 才配写入权限。
- [ ] **是否配了 3 类核心工具？** 文件系统（增删读写搜索）、浏览器（browser）、语言解释器（python/node）。95% 的任务只需要这些。
- [ ] **工具的接口心智是否与模型训练心智一致？** Unix 命令行 > 新协议。模型见过 `git commit` 几十亿次，没见过你的自定义 API。

---

## 判断框架优劣时的"双自洽"检验

- [ ] **自洽检验 1：这个框架的设计是否与模型运行原理自洽？** 模型是自回归的（自主决策下一步）→ 框架应该是"给能力"而非"预设流程"。如果你看到大量的状态图、预设流转路径、程序员指定的条件路由 → 不自洽。
- [ ] **自洽检验 2：这个框架是否与模型进步方向正交？** 如果模型变强 10 倍，这个框架是变得更有用还是变成瓶颈？如果框架需要"限制模型能力"才能工作 → 不正交，会被淘汰。

---

## 上下文管理时的 3 个警戒线

- [ ] **有没有随意修改 system prompt？** 每次修改 → prompt caching 失效 → KV cache 重新计算。最好的管理是"少管理"。
- [ ] **有没有随意裁剪历史对话？** 同上。优先用模型原生的上下文卸载策略，而非自己写截断逻辑。
- [ ] **长任务的"接棒机制"设计好了吗？** 上下文窗口满后，下一个 agent 如何接上？交接文档包含什么？（当前进度、环境状态、下一步计划）

---

## 评估是否真的需要多 Agent 时的冷静问题

- [ ] **这个问题单 Agent 能不能解决？** 先用单 agent 做 baseline。很多时候"多 agent"是过早优化。
- [ ] **不同 Agent 之间的信息是否真的隔离？** 如果一个 agent 能访问的信息另一个也能访问 → 多 agent 只是增加了延迟。
- [ ] **有没有用新路的"双问检验"？** (1) 这个 agent 做的决策能不能是一个函数调用？(2) 这个 agent 有的信息别的 agent 没有吗？

---

## PM 自我修养检验

- [ ] **我理解当前 SOTA 模型的能力边界吗？** 不知道模型能做什么、不能做什么，就判断不了 harness 是"不够"还是"过度"。
- [ ] **我在用"more context, less control"还是"more control, less context"思考产品？** 前者面向未来，后者面向过去。
- [ ] **我给 agent 配工具时，是先给最少的工具还是先给最多的？** 正确做法：先给最少（只读），按需升级。
