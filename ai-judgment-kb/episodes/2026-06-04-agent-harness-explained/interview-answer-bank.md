# Interview Answer Bank: Agent Harness 深度理解

**Source:** 十字路口 × 新路 | 2026.06.04
**Use for:** AI PM 面试、技术评审、内部汇报

---

## Q1: "你怎么理解 agent harness？为什么它重要？"

**问这个问题的人可能是：** 面试官考察你是否理解 agent 产品的工程层面

**30 秒版：**
Harness 就是"模型以外的一切"——工具、上下文管理、记忆、多 agent 编排。如果把模型比作大脑，harness 就是身体和机甲。现在 SOTA 模型之间的差距在缩小，harness 成为产品差异化的真正来源。一个 agent 产品能不能持续工作、能不能跨 session 记忆、能不能协调多个 agent，全看 harness。

**60 秒版（30 秒 + 三层架构）：**
Harness 分三层：第一层是执行能力——给模型配工具（CLI、文件系统、浏览器）；第二层是上下文状态——记忆、skills、上下文卸载；第三层是编排治理——多 agent 协调、权限控制、接力机制。判断一个 agent 产品成熟度的最快方法就是看它做到了第几层——只做第一层的是 demo，做到第二层的是产品，三层都做透的是平台。

**关键词：** harness, agent infrastructure, tool-use, context management, orchestration

**常见追问："那你怎么判断一个 harness 是好是坏？"**
→ 用"双自洽标准"：第一，是否与模型运行原理自洽（模型是自回归的，应该自主决策）；第二，是否与模型进步方向正交（模型越强，harness 不应成为瓶颈）。凡是需要"限制模型能力"才能工作的框架都会被淘汰。

## Q2: "你怎么看 MCP vs CLI 的争论？"

**问这个问题的人可能是：** 面试官在考察你对 agent 工具链的技术判断力

**30 秒版：**
CLI 的任务成功率普遍高于 MCP，这不是 CLI 设计得更好，而是模型训练语料的问题。Unix 命令在预训练中出现几十亿次，MCP 是近两年的新协议，语料占比不到 0.1%。所以给 agent 配工具时优先用命令行接口，这是"返璞归真"——不是造更多轮子，而是回到模型最熟悉的心智。

**60 秒版（30 秒 + 实践案例）：**
GitHub 就是一个很好的例子：GitHub CLI（`gh`）的任务成功率比 GitHub MCP 高很多。飞书 CLI 也比飞书之前的 MCP 插件效果更好。这不是协议设计的问题——MCP 在抽象上可能更优雅——而是模型对 Unix 命令的"肌肉记忆"远超任何新协议。作为 PM，这意味着你不需要追最新的工具协议，你应该优先选择模型"已经会用的"接口。

**关键词：** CLI-first, Unix philosophy, training data distribution, tool-use success rate

**常见追问："那 MCP 还有未来吗？"**
→ 如果 MCP 协议能被广泛采用，语料充分到模型"学会"它，那 CLI 的优势会缩小。但这个周期可能需要 2-3 年。现阶段务实的选择是 CLI。

## Q3: "你怎么判断一个 agent 产品是 demo 还是能落地的产品？"

**问这个问题的人可能是：** 面试官在考察你的产品判断力

**30 秒版：**
用三问判断：第一，它能处理超过上下文窗口的长任务吗？（不能就是 demo）第二，它有跨 session 的记忆和持续学习能力吗？（没有就是单次玩具）第三，它的工具链设计是"给模型自由"还是"限制模型"？（限制就是面向过去的设计）

**60 秒版（30 秒 + 具体检验维度）：**
更进一步，看三个信号：
1. **上下文管理是否"自洽"。** 好的产品尽量不做手动裁剪——每次裁剪都在破坏 prompt caching。如果一个产品到处是"截断历史""修改 system prompt"的逻辑，说明它没有真正理解模型的工作方式。
2. **工具链设计是否遵循"最少工具"原则。** 95% 的任务只需要文件系统、浏览器、语言解释器三类工具。如果一个产品配了 20+ 个工具，大概率是没想清楚。
3. **是否有"做梦"机制。** Cloud Code 有 auto-dream——定期后台整理记忆、纠错、合并。这听起来玄，但本质是"agent 能否自我迭代"。有这个机制的产品说明团队真的理解 agent 不是一次性工具。

**关键词：** production-readiness, context management, memory consolidation, tool design

## Q4: "做一个 AI 产品，模型、数据、harness，你觉得怎么排优先级？"

**问这个问题的人可能是：** 面试官在考察你的资源分配判断力

**30 秒版：**
模型排第一——agent 的本质是模型，模型不行后面都白做。上下文排第二——因为大部分人没有上千张卡去训练模型，我们能做的是上下文层面（记忆、skills、工具设计）。工具排第三——但配好工具可以显著放大模型能力，性价比很高。

**60 秒版（30 秒 + PM 视角）：**
从 PM 视角我要补充一点：这个排序是"技术上的优先级"，但"商业上的优先级"可能相反。如果市场上所有竞品都在用同一个 SOTA 模型，模型就不再是差异化的来源——harness 才是。就像所有手机都用高通芯片的时候，差异化来自工业设计、OS、生态。所以：**技术上模型 > 上下文 > 工具，商业上正好反过来。** 这就是为什么现在 harness 是创业热点。

**关键词：** prioritization, competitive moat, infrastructure vs application layer

## Q5: "你觉得 AI PM 和传统 PM 有什么不同？"

**30 秒版：**
传统 PM 的核心能力是理解用户需求、画好 UX/UI。今天的 AI PM 必须同时理解"技术周期到底在变什么"——因为 AI 产品本质上是在吃技术周期变化的红利。你不懂技术变化的内核，就做不出真正贴合这个变化的产品。

**60 秒版（30 秒 + 具体例子）：**
举个例子：如果你不理解"模型是自回归的"这一点，你就看不出为什么 LangChain 的 prompt flow 模式（预设每一步决策）会随着模型进步被淘汰。你会觉得"可控性和灵活性的权衡"只是一个工程偏好问题。但理解了模型的运行原理后你就知道——这不是偏好，这是方向问题。所以 AI PM 不需要会写代码，但需要理解模型怎么工作、harness 怎么设计的、工具链在哪个方向收敛。这决定了你能不能在技术评审中做出正确的产品判断。

**关键词:** technical fluency, technology cycle, product judgment, AI-native PM
