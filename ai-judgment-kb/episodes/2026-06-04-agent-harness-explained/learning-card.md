# Learning Card: Agent Harness — 模型以外的工程竞争力

**Episode:** 十字路口 × 新路（CRAI/1KB 创始人） | 2026.06.04
**Learning Goal:** AI 产品判断力

---

## Core Question

Agent 的上限到底由什么决定？当模型能力趋同，**harness（模型之外的一切工程实践）** 如何成为产品差异化的真正来源？

---

## 3 Core Judgments

**1. 模型才是 agent，其他都是 harness。**
🟡 Medium | 新路的工程实践 + Cloud Code 源码分析
→ 产品含义：选模型是第一决策，harness 是放大器而非替代品。模型不行，harness 再好也没用。

**2. Harness 好坏只有一个标准：是否与模型运行原理自洽，且与模型进步方向正交。**
🟡 Medium | 新路的工程实践总结
→ 产品含义：如果一个框架需要"限制模型能力"才能工作，它会在下一代模型发布时被淘汰。选工具链时问"它会不会随着模型变强而变强？"

**3. CLI > MCP，Unix 哲学是 agent 的原生接口。**
🟡 Medium | 新路的实际测试 + 预训练语料推理
→ 产品含义：给 agent 配工具时，优先用命令行而非新协议。预训练语料中 Unix 命令出现几十亿次，MCP 不到 0.1%。

---

## 3 Counter-Intuitive Insights

- **"最好的上下文管理是不做管理。"** 随意裁剪上下文会破坏 prompt caching，导致 KV cache 失效重新计算。你以为在优化，其实在浪费。
- **"产品经理的定义已经变了。"** 今天的 PM 不是在画 UX/UI，而是在理解技术周期变化的本质，把技术进步红利应用到场景中。
- **"MCP 是最近两年才提出的新协议，Unix 从 1971 年就存在了。"** 所以 CLI 的任务成功率比 MCP 高很多 — 不是 CLI 设计得好，是模型更熟悉它。

## Framework: Harness 三层架构

```
┌─────────────────────────────────┐
│  第三层：编排治理层               │
│  多 Agent 协调、权限控制、接力机制  │
├─────────────────────────────────┤
│  第二层：上下文状态层             │
│  Memory、Skills、上下文卸载/接棒   │
├─────────────────────────────────┤
│  第一层：执行能力层               │
│  CLI工具、文件系统、浏览器、解释器  │
└─────────────────────────────────┘
```

**决策规则：**
- 做单个 agent → 先配好第一层（工具），再加第二层（记忆）
- 做多 agent 系统 → 三层都需要，但第三层决定成败
- 评估竞品时 → 看它卡在哪一层。只做了第一层的是 demo，做到第二层的是产品，三层都做透的是平台

**When NOT to use:**
- 单次问答场景不需要完整的 harness
- 如果模型本身还不够强（任务成功率 < 60%），先换模型再优化 harness

## 5 Action Suggestions（AI PM 版）

1. **本周内：检查你产品的 agent 工具链。** 用了 MCP 的可以考虑迁移到 CLI，或至少做 A/B 对比测试
2. **下次技术评审时：问"这个设计给模型更多自由还是更多约束？"** 如果是后者，挑战它
3. **检查上下文管理策略：** 有没有随意修改 system prompt 或裁剪历史？每次这样做都在破坏 prompt caching
4. **用三层架构审视你的竞品：** 他们卡在哪一层？你可以在哪一层建立差异化？
5. **跟你的工程师聊"more context, less control"：** 这个原则他们认同吗？如果不认同，分歧在哪？

**参考答案：** 用"好 harness 的双自洽标准"来判断。LangChain 的 prompt flow 模式是"程序员预设每一步决策"→ 违背了"more context, less control"原则 → 它不与模型运行原理自洽（模型是自回归的，应该自主决策下一步）。更重要的是，它不与模型进步方向正交 — 模型越强，这种预设流程反而越会成为限制。新路的判断是：这套方法论"会越来越不适用"。你应该建议工程师先试"给模型配好工具，让它自己决策调用顺序"，作为 baseline 再对比。
