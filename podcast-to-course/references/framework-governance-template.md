# Framework Governance

Purpose: prevent the wiki from filling up with attractive but redundant named frameworks.

## Rule

Create a new framework only when it changes a decision rule.

If a new episode merely restates, reinforces, or gives another example of an existing decision rule, update the existing framework instead of creating a new one.

## Framework Admission Test

Before creating a new framework, answer all five:

1. **Decision delta:** What decision can the user make differently because this framework exists?
2. **Non-duplicate:** Which existing framework is closest, and why is this not just a rename?
3. **Boundary:** When should the user not apply it?
4. **Evidence:** What source anchors support it?
5. **Merge path:** If future episodes discuss the same idea, where should they attach?

If any answer is weak, do not create a new framework. Add a note to an existing concept or framework page.

## Relationship Labels

- **Creates**: introduces a genuinely new decision rule.
- **Reinforces**: same decision rule, new evidence.
- **Extends**: same decision rule, sharper boundary condition.
- **Contradicts**: credible source argues the opposite.
- **Renames only**: not allowed as a new framework.

## Maintenance Ritual

After every 5 episodes:

1. Count total frameworks.
2. Merge frameworks with overlapping decision rules.
3. Move weak frameworks into concept pages.
4. Add contradiction notes where credible sources disagree.
5. Update this inventory.
